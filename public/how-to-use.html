NAVADA

Executive Summary

NAVADA (New Analysis & Viability Assessment for Data-driven Analysis) is an AI-powered startup analysis platform designed to help investors, founders, and analysts make data-driven decisions about startup viability and risk.

The platform combines advanced data visualization, machine learning-powered intent detection, and an 8-factor viability scoring model to provide comprehensive startup analysis in seconds rather than hours.

Key Capabilities:

  • Interactive Dashboards: Plotly-powered charts with zoom, hover details, and click functionality
  • Memory & Personas: Session memory with Investor/Founder modes for personalized analysis
  • Auto-Generated Insights: AI-powered risk detection, opportunity identification, and actionable recommendations
  • Instant Chart Generation: 5 predefined visualizations + unlimited custom charts via natural language
  • AI-Powered Analysis: GPT-4 integration with conversation memory and intelligent Q&A
  • Viability Scoring: Comprehensive 8-factor assessment model (0-100 scale)
  • Founder Idea Benchmarking: Compare your startup metrics against dataset averages with percentile rankings
  • Portfolio Analysis: Multi-startup heatmap visualization with investment recommendations
  • Automated PDF Reports: One-click investment memos with executive summaries, charts, and recommendations
  • Custom Data Support: Upload and analyze your own startup datasets
  • Natural Language Interface: No technical expertise required - just ask questions

The Problem NAVADA Solves

Challenge 1: Time-Consuming Manual Analysis

Problem: Traditional startup analysis requires hours of spreadsheet work, manual chart creation, and repetitive calculations to assess viability factors like runway, burn rate, and market positioning.

Solution: NAVADA generates comprehensive visualizations and analysis in under 10 seconds through simple commands or natural language requests.

Challenge 2: Inconsistent Evaluation Criteria

Problem: Different investors and analysts use varying criteria to evaluate startups, leading to subjective assessments and missed risk factors.

Solution: NAVADA applies a standardized 8-factor viability model that weights critical success factors (runway, experience, market size, business model, moat, traction, growth, competition) to provide objective, reproducible scores.

Challenge 3: Technical Barriers to Data Visualization

Problem: Creating meaningful charts requires technical skills in tools like Python, R, or Excel, limiting access to data-driven insights for non-technical stakeholders.

Solution: Natural language chart generation allows anyone to request complex visualizations like "Show me a scatter plot of funding vs burn rate colored by sector" without writing a single line of code.

Challenge 4: Lack of Pattern Recognition

Problem: Identifying failure patterns across sectors, countries, and founder experience levels requires analyzing large datasets and recognizing subtle correlations.

Solution: AI-powered analysis detects patterns automatically and provides contextual insights like "UK startups have 71.4% failure rate due to lower average funding ($4.2M) vs successful startups ($11.3M)."

Who Benefits from NAVADA:

  • Investors & VCs: Rapid due diligence and portfolio risk assessment
  • Founders: Objective viability scoring to identify weaknesses before fundraising
  • Accelerators: Standardized evaluation criteria for cohort selection
  • Analysts: Automated data visualization for reports and presentations
  • Educators: Teaching entrepreneurship with real-world data and analysis

def fig_to_bytes(fig):
    """Convert matplotlib figure to PNG bytes."""
    buf = io.BytesIO()
    fig.savefig(buf, format='png', bbox_inches='tight', dpi=100)
    buf.seek(0)
    plt.close(fig)
    return buf.getvalue()

def plot_failure_timeline(df_in: pd.DataFrame):
    """Generate timeline bar chart."""
    fig, ax = plt.subplots(figsize=(12, 6))
    sorted_df = df_in.sort_values('Est_Failure_Year')
    colors = ['#ff4444' if f == 1 else '#44ff44' for f in sorted_df['Failed']]
    ax.barh(sorted_df['Startup'], sorted_df['Est_Failure_Year'], color=colors, alpha=0.7)
    ax.set_xlabel('Estimated Failure Year', fontsize=12)
    ax.set_title('Startup Failure Timeline', fontsize=14, fontweight='bold')
    plt.tight_layout()
    return fig_to_bytes(fig)

3.4 Create Viability Scoring Function

# Add to app.py:

def viability_score(features: dict) -> dict:
    """Calculate 8-factor viability score (0-100)."""

    # Extract features
    funding = features.get('funding', 3.0)
    burn = features.get('burn', 9.0)
    experience = features.get('experience', 3.0)
    market = features.get('market', 10.0)
    biz_model = features.get('biz_model', 3)
    moat = features.get('moat', 3)
    mrr = features.get('mrr', 0.0)
    growth = features.get('growth', 5.0)
    competition = features.get('competition', 3)

    # Calculate normalized components (0-1 scale)
    runway_months = (funding / burn) * 12
    c_runway = min(runway_months / 36, 1.0)  # 36 months = ideal
    c_exp = min(experience / 10, 1.0)  # 10 years = max
    c_market = min(market / 100, 1.0)  # $100B = max
    c_biz = biz_model / 5.0
    c_moat = moat / 5.0
    c_traction = min(mrr / 50, 1.0)  # $50k MRR = strong
    c_growth = min(growth / 20, 1.0)  # 20% = excellent
    c_comp = 1 - (competition / 5.0)  # Inverse scoring

    # Weighted score (0-100)
    weights = [0.18, 0.14, 0.14, 0.12, 0.10, 0.14, 0.12, 0.06]
    components = [c_runway, c_exp, c_market, c_biz, c_moat, c_traction, c_growth, c_comp]
    score = sum(w * c for w, c in zip(weights, components)) * 100

    return {
        "score": round(score, 1),
        "components": {
            "runway": c_runway,
            "experience": c_exp,
            "market": c_market,
            "business_model": c_biz,
            "moat": c_moat,
            "traction": c_traction,
            "growth": c_growth,
            "competition": c_comp
        },
        "survival_months": runway_months,
        "est_failure_year": FUNDING_YEAR + (runway_months / 12)
    }

Troubleshooting Common Issues

Issue 1: OpenAI API Errors

# Error: "Incorrect API key provided"
Solution: Check .env file has valid OPENAI_API_KEY
Verify: echo $OPENAI_API_KEY (Mac/Linux) or echo %OPENAI_API_KEY% (Windows)

# Error: "Rate limit exceeded"
Solution: Reduce request frequency or upgrade OpenAI plan

# Error: "Model not found"
Solution: Ensure using "gpt-4o-mini" (not "gpt-4o" or "gpt-4")

Issue 2: Chart Not Displaying

# Error: "Element.send() missing 1 required positional argument: 'for_id'"
Solution: Always attach images to parent message:
    msg = cl.Message(content="Title")
    await msg.send()
    image = cl.Image(content=png_bytes, name="chart.png", display="inline")
    await image.send(for_id=msg.id)  # ← Required!

# Error: Charts appear blank
Solution: Check matplotlib backend. Add to app.py:
    import matplotlib
    matplotlib.use('Agg')  # Non-interactive backend

Issue 3: Config Validation Errors

# Error: "header_links.0.icon_url Field required"
Solution: Add icon_url to all header links in config.toml:
    [[UI.header_links]]
        icon_url = "https://example.com/icon.png"

# Error: "show_disclaimer is not a valid config option"
Solution: Remove show_disclaimer line (not supported in Chainlit 2.8.0)

Issue 4: Module Import Errors

# Error: "No module named 'chainlit'"
Solution: Ensure virtual environment is activated:
    # Windows: venv\Scripts\activate
    # Mac/Linux: source venv/bin/activate
    pip install chainlit

# Error: "No module named 'openai'"
Solution: Install OpenAI package:
    pip install openai>=1.0.0

Security Best Practices

Critical Security Considerations:

  • Never commit .env: Add to .gitignore immediately
  • Use environment variables: Never hardcode API keys
  • Validate user input: Sanitize all file uploads and text inputs
  • Rate limiting: Prevent API abuse with request limits
  • HTTPS only: Always use SSL/TLS in production
  • Input validation: Check CSV file size, format, and contents
  • Error handling: Don't expose stack traces to users
  • Update dependencies: Regularly patch security vulnerabilities

Performance Optimization

Tips for Faster Response Times:

  • Cache chart images: Store generated PNGs for repeat requests
  • Lazy load data: Only load full dataset when needed
  • Optimize prompts: Shorter GPT-4 prompts = faster responses
  • Use async operations: Non-blocking I/O for API calls
  • Compress images: Reduce PNG file sizes with compression
  • Limit chart resolution: Use dpi=80 for faster generation
  • Batch API calls: Combine multiple requests where possible
